# Next-Level Feature Roadmap: Jazzapedia

*Generated: 2026-03-08*

## Current State Assessment

Jazzapedia is a remarkably well-built **music encyclopedia and discovery platform** tracking 5,479 artists and 28,034+ WWOZ track plays. Its strengths are clear: database-driven SSR via Astro, a cohesive "jazz club" design system (amber/brass/smoke palette, Playfair Display headings), and a standout GPU-instanced Three.js ecosystem graph rendering 3,500+ nodes at 60fps. The scraper pipeline is mature — automated WWOZ monitoring, Spotify matching, AI-generated artist profiles, and a multi-stage sync pipeline (SQLite → R2 → Git → D1) that runs daily.

**However, the platform has five specific architectural ceilings:**

1. **Rich data captured but invisible.** The database stores Spotify audio profiles (danceability, energy, valence, tempo), discography summaries, and 28k+ timestamped WWOZ plays — but none of this is visualized. Artist pages show text bios and a portrait; the data depth is hidden (`artists.audio_profile` JSON column, `artists.discography_summary`, `wwoz_tracks` table).

2. **Relationships are JSON strings, not a queryable graph.** Musical connections are stored as `artists.musical_connections` JSON with forward-only name strings — no reverse lookups, no multi-hop traversal, no aggregation. The 1.1MB `connections-index.json` is generated at build time, creating a 4+ hour lag for new connections and a scaling wall (10x artists = 10MB+ file).

3. **WWOZ data is archived but not analyzed.** 28k tracks with timestamps, shows, hosts, genres, and confidence scores exist but produce only a static insights page. No temporal analysis, no co-occurrence patterns, no host preference tracking, no genre trend lines.

4. **No geographic or temporal visualization.** Birth places, origins, career spans, and era data exist in the database but are rendered as plain text in the infobox. No map, no timeline, no era-based browsing.

5. **The graph is view-only.** The ecosystem graph is visually impressive but offers no analytical depth — no path-finding between artists, no cluster analysis, no "explore neighbors" interaction beyond single-click selection.

## Feature 1: WWOZ Intelligence Dashboard

### The Why

**Ceiling it breaks:** WWOZ data underutilization. 28,034 timestamped track plays with show names, hosts, genres, and Spotify match confidence — all sitting in `wwoz_tracks` — are reduced to a single static insights page generated by `compute-wwoz-insights.ts`.

**Market validation:** Radio airplay analytics is a proven market — [WARM](https://www.warmmusic.net/) monitors 23,000+ stations across 160 countries with detailed analytics dashboards. Music streaming platforms compete on discovery intelligence (Spotify's 70B daily data points for recommendations). Community radio listeners are underserved — no tool lets WWOZ fans explore *what's being played, when, and how it's changing*.

### The How

Build an interactive analytics dashboard at `/wwoz/insights` (replacing the current static page) with these views:

1. **Genre Heatmap** — Query `wwoz_tracks` grouped by `genres` × `date` to show genre play frequency over time. Surface rising/falling genres. SQL: `SELECT genres, strftime('%Y-%W', date) as week, COUNT(*) FROM wwoz_tracks GROUP BY genres, week`.

2. **Host Profiles** — Aggregate by `host` column: favorite genres, unique artists played, match rate. Each host gets a mini-page showing their musical personality.

3. **Co-occurrence Network** — Artists played on the same day/show cluster together. Query: `SELECT a.artist, b.artist, COUNT(*) FROM wwoz_tracks a JOIN wwoz_tracks b ON a.date = b.date AND a.artist < b.artist GROUP BY a.artist, b.artist HAVING COUNT(*) > 2`. This creates an *emergent* relationship graph from airplay data, independent of the curated connections.

4. **Discovery Velocity** — Track when artists first appear on WWOZ vs. when they were added to Jazzapedia. Surface "rising artists" (new to WWOZ in past 30 days).

**Files that change:**
- `apps/web/src/pages/wwoz/insights.astro` — Replace static rendering with interactive dashboard
- `apps/web/scripts/compute-wwoz-insights.ts` — Extend to pre-compute aggregations into new tables
- `packages/db/src/queries.ts` — Add analytics query functions
- New migration: `wwoz_artist_stats`, `wwoz_genre_trends`, `wwoz_host_profiles` materialized tables
- New components: `GenreHeatmap.astro`, `HostProfile.astro`, `DiscoveryTimeline.astro`

### The Tooling

**Pre-computed materialized tables** (not a new database) — run aggregation queries during the existing `sync-wwoz-db.ts` step and store results in summary tables. This keeps the D1/SQLite architecture intact while enabling fast dashboard queries. For the heatmap visualization, use **[Observable Plot](https://observablehq.com/plot/)** (lightweight, no React dependency, works perfectly as an Astro island with `client:load`). It's a better fit than D3 directly because Plot is declarative and produces SVG — ideal for SSR-first architecture where you want the chart to render without JS and enhance with interactivity.

## Feature 2: Artist Audio DNA & Comparison ✅ IMPLEMENTED (March 2026)

### The Why

**Ceiling it breaks:** Invisible rich data. Every artist with Spotify data has an `audio_profile` containing 7 numeric features (danceability, energy, valence, acousticness, instrumentalness, liveness, tempo) — stored but never displayed. The `discography_summary` (album/single counts, year range) is also hidden.

**Market validation:** Audio feature visualization is a proven engagement driver. Spotify's own "audio features" API powers dozens of third-party tools. [Every Noise at Once](https://everynoise.com/) built a cult following by visualizing genre-audio relationships. The 2026 trend is **contextual discovery** — apps using AI to surface "nuanced contextual discovery like identifying genres, moods, lyrical themes" ([Billboard/Universal-NVIDIA](https://www.billboard.com/pro/universal-nvidia-ai-music-discovery-fan-engagement-tools/)). No jazz-focused platform offers audio DNA comparison.

### The How

1. **Audio DNA Radar Chart** on each artist page — Render `audio_profile` as a radar/spider chart in the infobox (below portrait). Six axes: Danceability, Energy, Valence, Acousticness, Instrumentalness, Liveness. Tempo shown as a separate metric. Parse `artists.audio_profile` JSON (already in database).

2. **Artist Comparison Mode** — New route `/compare?a=miles-davis&b=john-coltrane`. Side-by-side layout showing overlaid radar charts, shared connections (intersection of `musical_connections`), shared genres, career timeline overlap. Link from artist pages via "Compare with..." button.

3. **"Similar Sound" Discovery** — Pre-compute audio profile similarity (Euclidean distance across 6 normalized features). Store top-10 similar artists per artist. Surface as "Sounds Like" section on artist detail pages. This creates a *data-driven* recommendation without needing user accounts.

**Files that change:**
- `apps/web/src/pages/artists/[...slug].astro` — Add AudioDNA component to infobox
- New: `apps/web/src/components/artist/AudioDNA.astro` — Radar chart (SVG, no JS needed for static render)
- New: `apps/web/src/pages/compare.astro` — Comparison page
- `apps/web/scripts/build-connections-static.ts` — Extend to pre-compute audio similarity
- New migration: `artist_similarity` table (artist_slug, similar_slug, distance)

### The Tooling

**SVG radar chart as a pure Astro component** — no charting library needed. A radar chart is 6 points on a polygon; it can be server-rendered as inline SVG with CSS transitions for hover effects. This keeps the zero-JS-by-default philosophy. For the comparison page overlay, use a single `<canvas>` island with `client:visible` for the animated overlay effect. Audio similarity computation runs in `build-connections-static.ts` during the existing prebuild step — no new infrastructure.

## Feature 3: Geographic & Temporal Atlas

### The Why

**Ceiling it breaks:** Geographic and temporal data exist but render as plain text. `birth_place`, `origin`, `career_span`, birth/death dates are all in the database. With 5,479 artists spanning jazz history from New Orleans to global, the geographic story is Jazzapedia's unique asset — no other platform maps the *geography of jazz influence*.

**Market validation:** [Music Galaxy](https://cprimozic.net/blog/building-music-galaxy/) and [Disco/graph](http://discograph.mbrsi.org/) prove that spatial music visualization drives engagement. The [Neo4j music knowledge graph](https://neo4j.com/blog/knowledge-graph/graphcast-building-a-music-knowledge-graph/) project explicitly highlights geographic and temporal dimensions as key graph features. Map-based exploration is standard in museum/archive UX but absent from music discovery tools.

### The How

1. **Artist Map** at `/map` — Plot artists by `origin` or `birth_place` on a world map. Cluster by city (New Orleans will dominate). Color by genre or era. Click cluster to expand, click artist to navigate. Geocode the ~200 unique city/country values in the database (one-time batch job, store lat/lng in a new `artist_locations` table).

2. **Timeline View** at `/timeline` — Horizontal scrollable timeline with artists placed by birth year. Vertical lanes by genre or instrument. Click to expand. Filter by genre preset (reuse graph presets from `graph-presets.json`). Career spans shown as horizontal bars.

3. **Era-Based Browsing** — Add era filters to `/artists` page: Pre-War, Swing Era, Bebop, Cool Jazz, Free Jazz, Fusion, Contemporary. Derive from `career_span` or birth decade. Surface in the existing alphabetical nav as a tab alternative.

**Files that change:**
- New: `apps/web/src/pages/map.astro` — Geographic visualization
- New: `apps/web/src/pages/timeline.astro` — Temporal visualization
- `apps/web/src/pages/artists/index.astro` — Add era filter tabs
- New migration: `artist_locations` table (slug, lat, lng, city, country)
- New: geocoding script in `apps/web/scripts/geocode-artists.ts`
- Navigation update in `Layout.astro` to add Map/Timeline links

### The Tooling

**[Leaflet](https://leafletjs.com/)** for the map — lightweight (42KB), no API key needed (uses OpenStreetMap tiles), works as an Astro island with `client:visible`. Leaflet is preferable to Mapbox GL for this use case: Jazzapedia is a personal/community project with no commercial budget, and Leaflet's marker clustering plugin handles the New Orleans density problem. For geocoding, use the free **[Nominatim API](https://nominatim.openstreetmap.org/)** (OpenStreetMap) in a one-time batch script — 200 unique cities is well within rate limits. The timeline can reuse the existing `graph-layout.json` pattern: pre-compute positions in a build script, render with lightweight canvas or SVG.

## Feature 4: Normalized Relationship Graph & Path Finding

### The Why

**Ceiling it breaks:** The fundamental graph data model. Currently, `artists.musical_connections` is a JSON blob of forward-only name strings. Reverse lookups require the 1.1MB `connections-index.json` built at compile time. Multi-hop traversal ("who connects Miles Davis to Herbie Hancock?") is impossible. At 10x scale, the static JSON approach fails entirely.

**Market validation:** [GATSY (Graph Attention Network for Music Artist Similarity)](https://arxiv.org/html/2311.00635v3) demonstrates that graph-based artist relationships are the frontier of music recommendation research. [GraphAware's music knowledge graph](https://graphaware.com/blog/building-and-exploring-music-knowledge-graph/) uses Neo4j with PageRank to find influential artists. The [Disco/graph](http://discograph.mbrsi.org/) project visualizes Discogs relationships interactively. Jazzapedia already has the visualization — it just needs the queryable data model underneath.

### The How

1. **Normalize relationships** into an `artist_relationships` table:
   ```sql
   CREATE TABLE artist_relationships (
     source_slug TEXT NOT NULL,
     target_slug TEXT NOT NULL,
     relationship_type TEXT NOT NULL, -- collaborator, influence, mentor, member, associated
     bidirectional INTEGER DEFAULT 0,
     PRIMARY KEY (source_slug, target_slug, relationship_type)
   );
   ```
   Populate from existing `musical_connections` JSON via a migration script. This makes reverse lookups a simple `WHERE target_slug = ?` query.

2. **Path finding API** — New endpoint `/api/graph/path?from=miles-davis&to=herbie-hancock` that finds shortest paths using BFS on the relationship table. SQLite recursive CTEs make this feasible: `WITH RECURSIVE path AS (...)`. Display as an interactive mini-graph on a new `/connect` page.

3. **Relationship analytics** — Pre-compute degree centrality (most connected artists), betweenness (bridge artists between communities), and cluster coefficients. Store in `artist_graph_stats` table. Surface as "Most Connected" and "Bridge Artists" sections on the graph page.

4. **Deprecate connections-index.json** — Once relationships are normalized, the static JSON becomes unnecessary for runtime queries. Keep it only for build-time wiki-link resolution (or migrate that too).

**Files that change:**
- New migration: `artist_relationships` table + `artist_graph_stats` table
- New: `apps/web/scripts/normalize-connections.ts` — One-time migration from JSON to relational
- `apps/web/scripts/build-connections-static.ts` — Refactor to read from `artist_relationships` table
- `apps/web/scripts/build-graph-layout.ts` — Read edges from table instead of JSON
- New: `apps/web/src/pages/api/graph/path.ts` — Path finding endpoint
- New: `apps/web/src/pages/connect.astro` — "Degrees of separation" page
- `packages/db/src/queries.ts` — Add relationship query functions

### The Tooling

**SQLite recursive CTEs** — No graph database needed. SQLite (and D1) support `WITH RECURSIVE` queries that can traverse relationships efficiently for the scale Jazzapedia operates at (5k-50k nodes). BFS path finding with a depth limit of 6 completes in <10ms on SQLite for this graph size. This avoids adding Neo4j or any new infrastructure — the existing D1/SQLite dual-deploy model continues to work. For graph analytics (centrality, clustering), compute at build time using the existing `d3-force` dependency and store results in the database.

## Feature 5: "What's Playing Now" Live Feed

### The Why

**Ceiling it breaks:** Real-time lag. The scraper captures WWOZ plays in near-real-time, but the website has a 4+ hour delay (scraper → Obsidian → daily sync → database → page). There's no way to see what WWOZ is playing *right now* through Jazzapedia.

**Market validation:** Live content is the #1 engagement driver in 2026 music apps. [KVPR's 2026 app refresh](https://www.kvpr.org/inside-kvpr/2026-01-30/kvpr-app-gets-new-features-new-look) highlights "easier access to live streams" as the top feature. Streaming platforms invest heavily in "listening rooms" and real-time social features. Community radio listeners want to know *what's playing now* and discover the artist instantly — this is Jazzapedia's unique value proposition that no other tool provides.

### The How

1. **Live ticker on homepage** — A horizontal scrolling bar showing the last 3-5 songs scraped from WWOZ. Artist names link to Jazzapedia profiles when they exist. Updates via Astro Server Islands (per-component server rendering with short cache).

2. **"Now on WWOZ" page** at `/live` — Current and recent tracks with artist cards, Spotify links, and match status. Show current DJ/program info. Uses Server Islands with 60-second cache (`cache: "max-age=60"`).

3. **Scraper → Database fast path** — The scraper already writes to SQLite immediately (`WorkflowService.ts`). Add a `wwoz_recent` table (or just query `wwoz_tracks ORDER BY date DESC, time DESC LIMIT 10`) that the web app reads via SSR. No WebSocket needed — Astro Server Islands with short TTL cache provide "near live" without architectural changes.

**Files that change:**
- `apps/web/src/pages/index.astro` — Add live ticker component
- New: `apps/web/src/pages/live.astro` — Live/recent tracks page
- New: `apps/web/src/components/LiveTicker.astro` — Server Island with 60s cache
- `apps/web/astro.config.mjs` — Enable Server Islands (experimental in Astro 5, stable in 6)
- Docker: Web container needs access to same SQLite that scraper writes to (already true via shared volume)

### The Tooling

**Astro Server Islands** — This is the key enabler. [Server Islands](https://docs.astro.build/en/guides/server-islands/) allow per-component server rendering with independent cache TTLs. The live ticker renders server-side every 60 seconds while the rest of the page caches for 5 minutes. No WebSocket, no client-side polling, no new infrastructure. For Cloudflare deployment, D1 queries replace SQLite — the scraper would need a new "fast sync" mode that pushes recent tracks to D1 more frequently (every 5 minutes via a lightweight cron, vs. the current daily batch).

## Priority & Sequencing

| Feature | Impact | Effort | Dependencies |
|---------|--------|--------|--------------|
| 1. WWOZ Intelligence Dashboard | High | Medium | None — builds on existing data |
| 2. Artist Audio DNA & Comparison | High | Low-Medium | None — data already in DB |
| 3. Geographic & Temporal Atlas | Medium-High | Medium | Geocoding batch job |
| 4. Normalized Relationship Graph | High (foundational) | Medium-High | Migration script, API changes |
| 5. "What's Playing Now" Live Feed | High (engagement) | Medium | Astro Server Islands, fast sync for CF |

**Recommended build order:**

1. **Audio DNA & Comparison** (Feature 2) — Lowest effort, highest immediate visual impact. Data already exists; it's primarily a UI build. Delivers a "wow" moment on every artist page.

2. **WWOZ Intelligence Dashboard** (Feature 1) — Next highest ROI. The data is there, the queries are straightforward, and it transforms Jazzapedia from "archive" to "intelligence platform." Creates content that drives return visits.

3. **Normalized Relationship Graph** (Feature 4) — Foundational for Features 3 and 5. Normalizing connections unblocks path finding, graph analytics, and eventually deprecating the 1.1MB static JSON. Do this before the atlas because the atlas benefits from normalized relationships.

4. **Geographic & Temporal Atlas** (Feature 3) — Builds on normalized relationships. Geocoding is a one-time batch job. The map and timeline are new pages that don't disrupt existing architecture.

5. **Live Feed** (Feature 5) — Last because it requires the most infrastructure change (Server Islands adoption, fast sync for Cloudflare). Also the most operationally complex — depends on scraper uptime for user-facing real-time features.
